## 🔄 标准工作流程 (SDLC)

### Phase 1: 需求分析 (PM主导)
```
PM 分析需求 → 编写 PRD → 定义验收标准 → 技术总监评审
```

### Phase 2: 架构设计 (Architect主导)
```
Architect 设计系统 → 定义 API → 技术总监评审 → PM确认
```

### Phase 3: 开发实现 (Engineer主导)
```
Engineer 编写代码 → 单元测试 → 自测 → 提交代码审查
```

### Phase 4: 测试验证 (QA主导)
```
QA 设计测试用例 → 执行测试 → Bug报告 → Engineer修复 → QA复测
```

### Phase 5: 部署上线 (Ops主导)
```
Ops 准备部署 → 预发布验证 → 生产部署 → 监控确认
```

### Phase 6: 验收总结 (Tech Lead主导)
```
技术总监验收 → PM确认 → 文档归档 → 复盘总结
```

## ✅ 质量门禁 (Quality Gates)

每个阶段必须通过才能进入下一阶段：

1. **需求门禁**: PRD 评审通过
2. **设计门禁**: 架构评审通过，技术风险可控
3. **代码门禁**: 
   - 代码审查通过
   - 单元测试覆盖率 > 80%
   - 无严重 Lint 错误
4. **测试门禁**: 
   - 所有 P0/P1 测试用例通过
   - 无阻塞性 Bug
5. **部署门禁**: 
   - 预发布环境验证通过
   - 回滚方案准备就绪

---

## 📋 原子化任务规范

每个任务必须满足：
- **单一职责**: 只做一件事
- **可验收**: 有明确的完成标准
- **可回滚**: 失败时不影响其他功能
- **有文档**: 代码+注释+文档三件套
- **有测试**: 单元测试覆盖核心逻辑
